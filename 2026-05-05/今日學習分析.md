## 概念鞏固：用自己的話解釋

**Q1：`ref` 包裝物件和直接用 `reactive`，行為上有什麼差異？**

**A：**
`ref({ name: 'Alice' })` 在內部會呼叫 `toReactive()`，因此物件部分和直接使用 `reactive` 行為一致。主要差異在於：

1. **存取方式**：`ref` 需要 `.value`，`reactive` 直接存取
2. **整體替換**：`ref.value = newObj` 可以整體替換並觸發更新；`reactive` 無法整體替換（只能修改屬性）
3. **解構安全性**：`ref` 解構出來的是 `Ref` 物件本身，仍有響應性；`reactive` 解構出的是普通值，失去響應性
4. **Composable 回傳**：推薦回傳 `ref`，讓使用方能安全解構

---

**Q2：`computed` 的「快取」到底快取了什麼？和普通函式相比，快取帶來了什麼效益？**

**A：**
`computed` 快取的是「上次執行 getter 的回傳值」。當 `_dirty = false`（依賴未改變），讀取 `.value` 直接回傳 `_value` 快取，不重新執行 getter 函式。

和普通函式的比較：
```typescript
// 普通函式：每次呼叫都執行
const getFiltered = () => tasks.value.filter(t => t.status === filter.value)
// 如果模板 render 10 次，這個函式執行 10 次

// computed：依賴未變則回傳快取
const filtered = computed(() => tasks.value.filter(t => t.status === filter.value))
// 模板 render 10 次，但只要 tasks 和 filter 沒變，getter 只執行 1 次
```

效益：在大型列表或複雜計算中，避免每次 render 都重複計算，顯著提升效能。

---

**Q3：為什麼 Composable 必須在 `<script setup>` 的同步執行期間呼叫？**

**A：**
Vue 3 的 Active Instance 機制：元件初始化時，Vue 設置一個全域指標 `currentInstance` 指向當前元件。所有 lifecycle hooks（`onMounted` 等）在呼叫時，會查詢 `currentInstance` 並把 hook 函式登記到對應元件。

如果 Composable 在 `setTimeout` 或 `await` 後才被呼叫，`currentInstance` 可能已變更（或為 null），lifecycle hooks 就無法正確綁定：

```typescript
// ❌ 錯誤：在非同步後呼叫 Composable
const setup = async () => {
  await fetchSomething()
  const { data } = useMyComposable()  // Active Instance 已不是當前元件！
}

// ✅ 正確：在同步期間呼叫
const { data } = useMyComposable()  // Active Instance 指向當前元件
const result = await fetchSomething()  // 非同步操作在 Composable 呼叫後
```
