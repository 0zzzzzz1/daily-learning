## 概念鞏固：W2 精華問答

**Q1：請說明 `watch + immediate: true` 和 `onMounted + watch` 的差異，並說明各自適用場景。**

**A：**
`onMounted + watch` 有兩個問題：（1）邏輯重複，初始載入和後續響應幾乎是相同程式碼，維護時需改兩處；（2）SSR 不友好，`onMounted` 在伺服器端不執行。

`watch + immediate: true` 的優點：（1）單一邏輯源，初始執行和後續響應走同一 callback；（2）在 setup 階段觸發，比 `onMounted` 更早；（3）搭配 `onCleanup` 更簡潔。

但有個例外：**需要操作 DOM 的場景仍必須用 `onMounted`**，因為 DOM 在 `onMounted` 前不存在。

---

**Q2：`watchEffect` 在非同步 callback 中的依賴追蹤有什麼限制？**

**A：**
```typescript
watchEffect(async () => {
  // ✅ await 之前：依賴被追蹤
  const id = userId.value

  await fetchSomething()  // ← 分界線

  // ⚠️ await 之後：依賴「不會」被追蹤
  console.log(otherRef.value)  // 改變 otherRef 不會重新觸發 watchEffect
})
```

原因：Vue 的依賴追蹤是同步的響應式系統。在遇到 `await` 後，程式碼控制權轉讓到 Event Loop，Vue 無法繼續追蹤後續的響應式存取。

解決方案：在 `await` 之前讀取所有需要追蹤的響應式資料，存入區域變數。

---

**Q3：Composable 為什麼可以在內部呼叫 `onMounted`？Vue 是如何知道要把 hook 綁定到哪個元件？**

**A：**
Vue 3 使用「Active Instance（活躍實例）」機制。當元件執行 `<script setup>` 時，Vue 會設置一個全域指標指向當前正在初始化的元件。當程式碼（無論是直接在 `<script setup>` 中，還是在被呼叫的 Composable 函式中）呼叫 `onMounted`、`onUnmounted` 等 API 時，Vue 會把這些 hook 註冊到當前 Active Instance。

這就是為什麼 Composable **必須在 `<script setup>` 同步呼叫期間**使用（不能在 `setTimeout` 或非同步後才呼叫），因為非同步後 Active Instance 可能已經不同了。
