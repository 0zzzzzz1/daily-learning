## 概念鞏固：用自己的話解釋

> 測試是否真正理解：能否用不看文件的方式，以自己的語言解釋核心概念？

**問：解構到底是一個語法糖，還是一個新機制？**

> 解構是**語法糖**。底層其實就是「逐一讀取屬性並賦值到新變數」，跟手寫 `const a = obj.a; const b = obj.b;` 完全等價。理解這件事，就能推導出解構的所有行為——包含為什麼會失去 Proxy 響應性（因為變數只記住讀取當下的值）、為什麼 `null` 不觸發預設值（因為 `null !== undefined`，屬性明確被賦予 null）。

**問：Spread 和 Rest 語法長得一樣，要怎麼分？**

> 看「位置」。在 **右側/值的位置** 是 Spread（打散一個集合），在 **左側/宣告的位置** 是 Rest（蒐集剩下的）。範例：`[...a, ...b]` 是 Spread、`const [first, ...rest] = arr` 是 Rest；`fn(...args)` 是 Spread、`function fn(...args)` 是 Rest。

**問：`{ ...obj }` 是深拷貝還是淺拷貝？為什麼？**

> **淺拷貝**。它只複製物件的第一層屬性值，如果屬性本身是物件/陣列（參考型別），複製的只是「同一個參考」。修改巢狀物件時，原物件也會被改。要深拷貝，用 `structuredClone(obj)`（現代瀏覽器原生支援，也能處理 Map/Set/循環參考）。

**問：為什麼 `toRefs` 能解決解構失去響應性的問題？**

> 因為 `toRefs` 把原本 reactive 物件的每個**基本型別屬性**，分別包成 `ref` 物件（`{ value: ... }`）。ref 本身是**物件（參考型別）**，解構後新變數拿到的是 ref 物件的參考，修改 `.value` 會觸發原 reactive 物件對應屬性的 `set`，所以響應性被保留。
