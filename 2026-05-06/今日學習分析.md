## 概念鞏固：用自己的話解釋

**Q1：為什麼 `<template>` 中的 `{{ count }}` 不需要 `.value`，但 `{{ obj.count }}` 卻可能需要？**

**A：**
模板編譯器只會對「頂層識別字」做 auto-unwrap。當你寫 `{{ count }}`，編譯器看到 `count` 是一個 ref（透過 setup 回傳的對應檢查），會自動編譯成 `count.value`。

但 `{{ obj.count }}` 編譯時只看到 `obj` 是頂層識別字，編譯器不會深入檢查 `obj.count` 是否為 ref——因為它做不到（屬性訪問是執行期才確定的型別）。所以如果 `obj.count` 是一個 ref，模板裡仍要寫 `obj.count.value`。

**例外**：如果 `obj` 本身是 reactive 物件，那 `obj.count`（即使是 ref）會被 reactive 的 Proxy 內部自動解包——這是 reactive 的特性，不是模板的特性。

---

**Q2：為什麼 Composable 推薦使用 `reactive + toRefs` 而不是直接寫多個 `ref`？**

**A：**
兩者最終效果一致，差異在「撰寫舒適度」與「對外介面」：

| 比較 | 直接寫多個 ref | reactive + toRefs |
|------|:-------------:|:-----------------:|
| Composable 內部寫起來 | 每個變數要 `.value` | 直接 `state.x = y` |
| 對外回傳 | `{ a, b, c }` | `{ ...toRefs(state) }` |
| 對外解構 | 兩者皆可安全解構 | ✅ 安全 |
| 執行成本 | 多個 RefImpl | 1 個 Proxy + N 個 toRef |

實務選擇：**狀態多而相關**用 reactive+toRefs；**獨立小變數**用 ref。

---

**Q3：什麼情況下會選 `shallowRef` 而不是 `ref`？**

**A：**
三個明確訊號：

1. **資料只整體替換**：例如 API 回傳的整批資料，每次都是新陣列，不會局部修改。
2. **資料巨大**：上千筆物件、地理座標、影像資料。深度 `Proxy` 化會在初始化和讀取時造成效能成本。
3. **內部是非 Vue 管轄的物件**：例如 Mapbox 實例、Three.js 場景、Monaco Editor——讓 Vue 不要去代理它們的內部。

實務範例：
```typescript
// API 整批分頁資料
const tasks = shallowRef<Task[]>([])
tasks.value = await api.list()   // ✅ 整批替換觸發
// tasks.value.push(newTask)     // ❌ 不會觸發，需要：
tasks.value = [...tasks.value, newTask]  // ✅ 整體替換
```

│   ├── markRaw：永久標記不被響應化
│   ├── toRaw：取出 Proxy 前的原始物件

    ├── 物件 + 只關心第一層 → shallowReactive
