## 學習前 vs 學習後對照

### `ref` 與 `reactive` 的理解

| | 學習前 | 學習後 |
|---|--------|--------|
| **理解程度** | 知道兩個都能做響應式，但不知道差在哪 | 清楚理解各自適用場景與底層差異 |
| **使用習慣** | 隨機選擇，常用 `reactive` 包一切 | 基本型別 → `ref`；物件群組 → `reactive` |
| **陷阱意識** | 不知道解構會失去響應性 | 已知解構陷阱，會使用 `toRefs` 解決 |
| **`.value` 原因** | 覺得這個設計很奇怪 | 理解這是 Proxy 無法攔截基本型別而必要的包裝 |

### `computed` 的理解

| | 學習前 | 學習後 |
|---|--------|--------|
| **理解程度** | 知道是「算出來的值」，偶爾用 | 理解快取機制、何時重新計算、與 method 的效能差異 |
| **使用時機** | 不確定時用 method 取代 | 衍生資料 → `computed`；有副作用或不需快取 → method |

### `watch` 與 `watchEffect` 的理解

| | 學習前 | 學習後 |
|---|--------|--------|
| **理解程度** | 只知道 `watch`，不知道 `watchEffect` | 清楚兩者差異，知道各自的使用時機 |
| **惰性執行** | 不知道 `watch` 預設是惰性的 | 已知，會根據需求加 `{ immediate: true }` |

### Proxy 底層的理解

| | 學習前 | 學習後 |
|---|--------|--------|
| **理解程度** | 「黑魔法，改了資料畫面就更新了」 | 理解 `get` 攔截做依賴收集（track），`set` 攔截做通知更新（trigger） |
| **與 Vue 2 差異** | 不清楚 Vue 2 / Vue 3 的響應式差在哪 | Vue 2 用 `Object.defineProperty`（無法監聽新增屬性）；Vue 3 用 `Proxy`（全面攔截） |

## 概念鞏固：用自己的話解釋

> 測試是否真正理解：能否用不看文件的方式，以自己的語言解釋核心概念？

**問：Vue 3 為什麼改用 Proxy？**

> Vue 2 的 `Object.defineProperty` 只能針對**已存在**的屬性設定 getter/setter，所以在物件上**新增屬性**或刪除屬性時，Vue 2 無法偵測到，必須用 `Vue.set()` 補救。Proxy 則是攔截整個物件的所有操作（包含新增屬性），解決了這個根本限制，也讓陣列的響應性更自然。

**問：`ref` 為什麼需要 `.value`？**

> JavaScript 的基本型別（number、string、boolean）是**值傳遞**，不是參考傳遞，所以 Proxy 無法直接包裝它們。Vue 3 的做法是把基本型別放進一個物件 `{ value: ... }` 再用 Proxy 包裝，這樣就能偵測到 `.value` 的讀寫。這是底層的必要設計，不是「多此一舉」。

**問：什麼時候用 `watch`，什麼時候用 `watchEffect`？**

> 如果我**明確知道要監聽哪個資料**，而且需要拿到**變化前後的值做比較**，用 `watch`。如果我只是想「每當某些資料改變就執行某件事，不在乎是哪個資料觸發的」，用 `watchEffect`——它會自動追蹤函式內用到的所有響應式資料，比較省事但彈性少。
