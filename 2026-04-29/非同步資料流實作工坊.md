## 模組一：`createHttpClient` — HTTP 客戶端

**設計思維**：為什麼不直接用 `fetch`？

- `fetch` 分散在各元件中，`baseUrl` 重複
- 狀態碼錯誤處理不統一（`fetch` 不會對 4xx/5xx 拋出錯誤！）
- 無法集中加入 header（如 Authorization）

**關鍵知識點**：`fetch` 只對「網路錯誤」拋出例外，HTTP 4xx/5xx 的 `response.ok` 為 `false` 但不拋出錯誤。必須手動檢查。

```typescript
// httpClient.ts

// 自訂錯誤類別，攜帶 HTTP 狀態碼
export class HttpError extends Error {
  constructor(
    public statusCode: number,
    message: string
  ) {
    super(`[HTTP ${statusCode}] ${message}`)
    this.name = 'HttpError'
  }
}

export interface HttpClient {
  get<T>(path: string): Promise<T>
  post<T, B = unknown>(path: string, body: B): Promise<T>
  put<T, B = unknown>(path: string, body: B): Promise<T>
  delete<T>(path: string): Promise<T>
}

export const createHttpClient = (baseUrl: string): HttpClient => {
  // 共用的請求邏輯（DRY 原則）
  const request = async <T>(
    path: string,
    options: RequestInit = {}
  ): Promise<T> => {
    const response = await fetch(`${baseUrl}${path}`, {
      headers: {
        'Content-Type': 'application/json',
        ...options.headers,
      },
      ...options,
    })

    // ⚠️ fetch 不對 4xx/5xx 拋出錯誤，需手動檢查
    if (!response.ok) {
      const message = await response.text().catch(() => response.statusText)
      throw new HttpError(response.status, message)
    }

    // 204 No Content — 沒有回應本體
    if (response.status === 204) {
      return undefined as T
    }

    return response.json() as Promise<T>
  }

  return {
    get: <T>(path: string) =>
      request<T>(path, { method: 'GET' }),

    post: <T, B = unknown>(path: string, body: B) =>
      request<T>(path, { method: 'POST', body: JSON.stringify(body) }),

    put: <T, B = unknown>(path: string, body: B) =>
      request<T>(path, { method: 'PUT', body: JSON.stringify(body) }),

    delete: <T>(path: string) =>
      request<T>(path, { method: 'DELETE' }),
  }
}
```

## 模組二：`withRetry` — 請求重試裝飾器

**設計思維**：什麼是「裝飾器模式（Decorator Pattern）」？

> 裝飾器接受一個函式，回傳一個「功能增強版」的函式。不修改原函式，只是包裝它。

**兩種實作方式對比**：

```typescript
// retry.ts

export interface RetryOptions {
  maxRetries?: number  // 預設 3 次
  delay?: number       // 預設 1000ms
  onRetry?: (attempt: number, error: Error) => void  // 可選：重試時的回調
}

// 方式一：遞迴（優：簡潔；缺：deep stack）
const withRetryRecursive = <T>(
  fn: () => Promise<T>,
  options: RetryOptions = {}
): (() => Promise<T>) => {
  const { maxRetries = 3, delay = 1000, onRetry } = options

  const attempt = async (retriesLeft: number): Promise<T> => {
    try {
      return await fn()
    } catch (error) {
      if (retriesLeft <= 0) throw error  // 超過重試次數，直接拋出
      
      onRetry?.(maxRetries - retriesLeft + 1, error as Error)
      
      // 等待 delay 毫秒後重試
      await new Promise(resolve => setTimeout(resolve, delay))
      return attempt(retriesLeft - 1)
    }
  }

  return () => attempt(maxRetries)
}

// 方式二：迴圈（優：無堆疊問題；缺：稍微複雜）
export const withRetry = <T>(
  fn: () => Promise<T>,
  options: RetryOptions = {}
): (() => Promise<T>) => {
  const { maxRetries = 3, delay = 1000, onRetry } = options

  return async (): Promise<T> => {
    let lastError: Error

    for (let attempt = 0; attempt <= maxRetries; attempt++) {
      try {
        return await fn()
      } catch (error) {
        lastError = error as Error
        
        // 最後一次嘗試失敗，不等待直接拋出
        if (attempt === maxRetries) break
        
        onRetry?.(attempt + 1, lastError)
        await new Promise(resolve => setTimeout(resolve, delay))
      }
    }

    throw lastError!
  }
}
```

**使用範例**：

```typescript
const client = createHttpClient('https://api.example.com')

const fetchWithRetry = withRetry(
  () => client.get<Property>('/properties/1'),
  {
    maxRetries: 3,
    delay: 1000,
    onRetry: (attempt, error) => {
      console.warn(`第 ${attempt} 次重試，原因：${error.message}`)
    }
  }
)

const property = await fetchWithRetry()
```

**自我檢核**：
- [ ] 為何 `maxRetries = 3` 但迴圈的條件是 `attempt <= maxRetries`（共執行 4 次）？
- [ ] 如果重試間隔需要「指數退避（Exponential Backoff）」，如何修改 `delay`？
  - 提示：`delay * Math.pow(2, attempt)`

## 模組三：`withCache` — 請求快取裝飾器

**設計思維**：為何需要快取？快取的邊界在哪？

> 相同的 API 請求在短時間內不需要重複發送。但快取必須有「失效機制（TTL）」，否則拿到過期資料。

```typescript
// cache.ts

interface CacheEntry<T> {
  data: T
  expiredAt: number  // timestamp (ms)
}

export const withCache = <TArgs extends unknown[], TResult>(
  fn: (...args: TArgs) => Promise<TResult>,
  ttl: number = 5 * 60 * 1000  // 預設 5 分鐘
): ((...args: TArgs) => Promise<TResult>) => {
  // 每個 withCache 呼叫有自己獨立的快取空間
  const cache = new Map<string, CacheEntry<TResult>>()

  return async (...args: TArgs): Promise<TResult> => {
    // 使用參數的 JSON 序列化字串作為 key
    const key = JSON.stringify(args)

    // 檢查快取是否存在且未過期
    const cached = cache.get(key)
    if (cached && Date.now() < cached.expiredAt) {
      console.log(`[Cache HIT] key: ${key}`)
      return cached.data
    }

    console.log(`[Cache MISS] key: ${key}`)

    // 快取不存在或已過期，發出真實請求
    const data = await fn(...args)

    // 儲存到快取
    cache.set(key, {
      data,
      expiredAt: Date.now() + ttl
    })

    return data
  }
}
```

**邊界情況思考**：

```typescript
// ⚠️ JSON.stringify 的限制：
JSON.stringify([undefined]) // "[null]"  ← undefined 被轉成 null
JSON.stringify([() => {}])  // "[null]"  ← 函式被忽略

// ⚠️ 物件屬性順序：
JSON.stringify({ a: 1, b: 2 }) // '{"a":1,"b":2}'
JSON.stringify({ b: 2, a: 1 }) // '{"b":2,"a":1}'  ← 不同的 key！

// 因此 withCache 最適合「基本型別參數」或「順序固定的物件」
```

**自我檢核**：
- [ ] 若兩個元件同時呼叫 `cachedFetch(1)`，且兩次都 cache miss，會發出幾次請求？如何解決？
  - 這就是「快取穿透（Cache Stampede）」問題 → 進階延伸題

## 模組四：`fetchConcurrent` — 並行請求控制器

**設計思維**：為何需要控制並行數量？

> `Promise.all` 是「全部同時發出」，若有 100 個請求同時發出，會打爆後端。需要「批次處理」— 每次最多發 N 個，前一批完成後才發下一批。

**關鍵演算法**：Sliding Window（滑動窗口）

```typescript
// concurrent.ts

export const fetchConcurrent = async <T>(
  requests: Array<() => Promise<T>>,
  limit: number = 3
): Promise<T[]> => {
  const results: T[] = new Array(requests.length)
  let currentIndex = 0  // 追蹤下一個要執行的請求索引

  // 單一 worker：持續取任務直到沒有為止
  const worker = async (): Promise<void> => {
    while (currentIndex < requests.length) {
      const index = currentIndex++  // 取得自己的任務索引（注意：需先取再遞增）
      results[index] = await requests[index]()
    }
  }

  // 啟動 limit 個並行 worker
  const workers = Array.from({ length: Math.min(limit, requests.length) }, () => worker())
  await Promise.all(workers)

  return results
}
```

**視覺化理解**：

```
假設 8 個請求，limit = 3

時間軸 →
Worker 1: [req1]...[req4]...[req7]...
Worker 2: [req2]...[req5]...[req8]...
Worker 3: [req3]...[req6]...

每個 worker 完成一個請求後，立即從剩餘的「待辦清單」取下一個
結果陣列順序永遠對應輸入順序（透過 index 儲存）
```

**自我檢核**：
- [ ] 為什麼 `const index = currentIndex++` 而不是 `const index = ++currentIndex`？
- [ ] 若某一個請求拋出錯誤，其他 worker 會怎樣？（目前實作的行為）
- [ ] 如果要改成「失敗繼續（不因單一失敗中止）」，要怎麼修改？

## 模組五：`useAsyncState` — Vue 3 Composable

**設計思維**：為何要包裝成 Composable？

> 每個需要非同步資料的元件都要管理 `isLoading`、`error`、`data` 三個狀態，重複程式碼。Composable 讓狀態邏輯可以被複用。

```typescript
// useAsyncState.ts
import { ref, type Ref } from 'vue'

export interface AsyncState<T> {
  data: Ref<T | null>
  isLoading: Ref<boolean>
  error: Ref<string | null>
  execute: () => Promise<void>
}

export const useAsyncState = <T>(
  fn: () => Promise<T>
): AsyncState<T> => {
  const data = ref<T | null>(null) as Ref<T | null>
  const isLoading = ref<boolean>(false)
  const error = ref<string | null>(null)

  const execute = async (): Promise<void> => {
    isLoading.value = true
    error.value = null

    try {
      data.value = await fn()
    } catch (e) {
      error.value = e instanceof Error ? e.message : '未知錯誤'
      data.value = null
    } finally {
      isLoading.value = false
    }
  }

  return { data, isLoading, error, execute }
}
```

**在元件中使用**：

```vue
<script setup lang="ts">
import { onMounted } from 'vue'
import { useAsyncState } from './async-utils'
import type { Property } from './types'

const client = createHttpClient('https://api.example.com')

const { data: properties, isLoading, error, execute } = useAsyncState<Property[]>(
  () => client.get('/properties')
)

onMounted(execute)
</script>

<template>
  <div v-if="isLoading">載入中...</div>
  <div v-else-if="error">錯誤：{{ error }}</div>
  <ul v-else>
    <li v-for="p in properties" :key="p.id">{{ p.title }}</li>
  </ul>
</template>
```

## 手動測試案例（`作業實作/test.ts`）

```typescript
// test.ts — 在瀏覽器 console 或 Node.js 中執行

// ── Mock fetch（讓測試不需要真實後端）──────────────────────────────
let callCount = 0

const mockFetch = (shouldFail = false, delay = 100) =>
  new Promise<{ id: number; title: string }>((resolve, reject) => {
    callCount++
    setTimeout(() => {
      if (shouldFail) {
        reject(new Error('模擬網路錯誤'))
      } else {
        resolve({ id: callCount, title: `資料 #${callCount}` })
      }
    }, delay)
  })

// ── 測試 withRetry ─────────────────────────────────────────────────
console.group('withRetry 測試')

// 案例一：成功不需重試
callCount = 0
const fetchSuccess = withRetry(() => mockFetch(false), { maxRetries: 3 })
const result1 = await fetchSuccess()
console.assert(callCount === 1, '成功時應只呼叫 1 次')
console.log('案例一通過：', result1)

// 案例二：失敗後重試 3 次
callCount = 0
let retryCount = 0
const fetchFail = withRetry(
  () => mockFetch(true),
  { maxRetries: 3, delay: 10, onRetry: () => retryCount++ }
)
try {
  await fetchFail()
} catch (e) {
  console.assert(callCount === 4, `應呼叫 4 次（1次原始 + 3次重試），實際：${callCount}`)
  console.assert(retryCount === 3, `應重試 3 次，實際：${retryCount}`)
  console.log('案例二通過：正確重試並最終拋出錯誤')
}

console.groupEnd()

// ── 測試 withCache ─────────────────────────────────────────────────
console.group('withCache 測試')

callCount = 0
const cachedFetch = withCache(
  (id: number) => mockFetch(false, 50),
  1000  // 1 秒 TTL
)

// 案例一：相同參數，第二次應命中快取
const r1 = await cachedFetch(1)
const r2 = await cachedFetch(1)
console.assert(callCount === 1, `相同參數應只呼叫 1 次，實際：${callCount}`)
console.log('案例一通過：快取命中')

// 案例二：不同參數，各自發出請求
const r3 = await cachedFetch(2)
console.assert(callCount === 2, `不同參數應呼叫 2 次，實際：${callCount}`)
console.log('案例二通過：不同參數各自請求')

console.groupEnd()

// ── 測試 fetchConcurrent ───────────────────────────────────────────
console.group('fetchConcurrent 測試')

callCount = 0
const ids = [1, 2, 3, 4, 5]
const requests = ids.map(id => () => mockFetch(false, Math.random() * 100))

const results = await fetchConcurrent(requests, 2)  // 最多 2 個並行

console.assert(results.length === 5, `應有 5 個結果，實際：${results.length}`)
console.assert(callCount === 5, `應呼叫 5 次，實際：${callCount}`)
console.log('測試通過：', results)

console.groupEnd()
```

### 挑戰一：快取穿透防護（Request Deduplication）

```typescript
// 進階版 withCache：防止多個同時請求 miss cache 後發出重複請求
// 提示：用 Map 儲存「進行中的 Promise」

const pendingRequests = new Map<string, Promise<TResult>>()

// 當 cache miss 時，先檢查是否有相同 key 正在進行
const existing = pendingRequests.get(key)
if (existing) return existing  // 複用進行中的請求

// 儲存當前 Promise
const promise = fn(...args).finally(() => pendingRequests.delete(key))
pendingRequests.set(key, promise)
return promise
```
