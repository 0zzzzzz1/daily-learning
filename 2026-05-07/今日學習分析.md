## 概念鞏固：用自己的話解釋

**Q1：為什麼 `useTasks` 的 `tasks` 用 `ref<Task[]>` 而不是 `reactive({ list: Task[] })`？兩者的 trigger 機制有何不同？**

**A：**

`ref<Task[]>` 的 trigger 機制是：`.value` 的位址改變時觸發。所以每次操作都要整批替換：
```typescript
tasks.value = [...tasks.value, newTask]  // 新陣列 → 位址改變 → 觸發更新
```

`reactive({ list: [] })` 的 trigger 機制是：Proxy 代理了陣列的 `push`、`splice` 等方法，攔截 set 操作後自動 trigger。所以可以直接 mutation：
```typescript
state.list.push(newTask)  // Proxy 攔截 → 觸發更新
```

選 `ref` 的理由：
1. **語義清晰**：每次修改都明確說「這是一個新的任務列表」，不可變更新讓狀態可預測
2. **TypeScript 型別乾淨**：`ref<Task[]>` 的 `.value` 是 `Task[]`，比 `reactive` 的巢狀型別推導更直觀
3. **未來對 shallowRef 的相容**：如果任務列表很大，只需把 `ref` 換成 `shallowRef` 即可優化，不需重構邏輯

---

**Q2：今日設計的 `useTasks`，其 `return` 沒有使用 `toRefs`。什麼情況下才需要 `toRefs`？**

**A：**

`toRefs` 的用途是：把 **`reactive` 物件**的屬性轉成 ref，讓解構後仍保持響應性。

```typescript
// 需要 toRefs 的情況
const state = reactive({ count: 0, name: 'task' })
return { ...toRefs(state) }  // ✅ 解構後 count 和 name 仍是連動 ref

// 不需要 toRefs 的情況
const count = ref(0)
const name = ref('task')
return { count, name }  // ✅ 本身就是 ref，直接回傳即可
```

`useTasks` 的所有狀態（`tasks`、`filter`、`newTaskText`）本來就是 `ref`，直接回傳就是 ref，解構後自然保持響應性。`computed` 回傳的也是 ref（ComputedRef）。
所以不需要 `toRefs`。

---

**Q3：`filteredTasks` 定義為 `computed`，而非普通函式在模板中呼叫。能解釋快取機制帶來的效能差異嗎？**

**A：**

```typescript
// computed（有快取）
const filteredTasks = computed(() => tasks.value.filter(t => !t.done))
```

`computed` 內部的 `_dirty flag`（W3 Day 1 所學）：
- 初次讀取：執行 getter，計算結果，設 `_dirty = false`
- 依賴（`tasks`、`filter`）沒變：再次讀取直接回傳快取，getter **不執行**
- 依賴改變：設 `_dirty = true`，下次讀取才重新執行 getter

```typescript
// 普通函式（無快取）
const getFilteredTasks = () => tasks.value.filter(t => !t.done)
```

在模板中 `v-for="task in getFilteredTasks()"` → 每次元件 re-render（包括其他無關資料改變）都會執行一次 `.filter()`。如果有 1000 筆任務，每次 re-render 都要遍歷 1000 筆。

`computed` 只在 `tasks` 或 `filter` 真正改變時才重算，其他情況直接讀快取。
